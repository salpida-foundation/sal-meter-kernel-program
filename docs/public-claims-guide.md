# Public Claims Guide — How to Describe the Project Without Distortion

> This page explains how the Sal-Meter Kernel Program should and should not be described in public-facing language.  
> It is an operational helper for repository readers, labs, contributors, and observers.  
> It does not replace canonical DOI-registered claims-control or publication-boundary documents.

---

## 1. Why This Page Exists

Projects like this are often damaged not only by technical weakness, but by language drift.

A project may still be structurally early, yet public language can make it sound:

- fully proven
- clinically meaningful
- officially compliant
- commercially ready
- already open to broad participation
- authorized for naming and branding
- more settled than it actually is

This page exists to prevent that.

Its purpose is simple:

**to help readers speak accurately about the project without overstating its current stage, authority, or meaning.**

---

## 2. The Core Rule

The safest rule is this:

**Describe the project as a pre-opening, research-stage, kernel-first program.  
Do not describe it as a completed technology, certified system, or clinically validated platform.**

This one rule eliminates most avoidable distortion.

---

## 3. What the Project May Be Described As

At the current stage, the project may be described as:

- a **research-stage program**
- a **pre-opening kernel program**
- a **CAIS-aligned measurement research effort**
- a **molecular-electrochemical state-separability program**
- a **bounded exploratory interface program**
- a **kernel-first validation effort**
- a **non-diagnostic, non-therapeutic research architecture**
- a **structured attempt to test whether a reproducible signal interface can be made real**

These descriptions are careful, stage-accurate, and operationally useful.

---

## 4. What the Project Should Not Be Described As

At the current stage, the project should **not** be described as:

- a proven consciousness meter
- a clinically validated device
- a diagnostic instrument
- a therapeutic platform
- an approved or certified system
- a commercially released biosensor
- an officially open public competition
- a finalized SDK
- a universal proof that consciousness has been measured
- an unrestricted public brand anyone may use

These phrases create false certainty and confuse the project’s current status.

---

## 5. Safe Core Language

When speaking publicly, use language like this:

### Safe examples

- “The Sal-Meter Kernel Program is a pre-opening research-stage effort.”
- “The project is testing whether a bounded molecular-electrochemical interface can support reproducible signal separation.”
- “At this stage, the work is non-diagnostic and non-therapeutic.”
- “GitHub is being used as an operational and preparatory layer, not as the canonical authority layer.”
- “The broader opening has not yet occurred.”
- “The current burden is kernel proof, not scale.”
- “The project is structured around Layer-0, internal phases, and lock conditions before any broader opening.”

These phrases are strong without overstating.

---

## 6. Unsafe Core Language

Avoid language like this:

### Unsafe examples

- “This device measures consciousness.”
- “This is the first proven consciousness sensor.”
- “This is already clinically meaningful.”
- “This system is certified.”
- “The competition is open.”
- “This repository authorizes builders to use the Sal-Meter name.”
- “This proves consciousness scientifically.”
- “This is ready for market.”
- “This repository confirms compliance.”
- “This is an approved medical platform.”

These phrases outrun the project.

---

## 7. The Correct Description of Current Status

If someone asks, “What is it, right now?” the strongest short answer is:

**The Sal-Meter Kernel Program is a pre-opening, research-stage, non-diagnostic, non-therapeutic effort to determine whether a bounded CAIS-aligned molecular-electrochemical interface can produce reproducible state-separable signals under disciplined conditions.**

That sentence is long, but structurally safe.

A shorter version is:

**It is a kernel-first research program, not yet a broadly open competition or certified system.**

---

## 8. The Correct Description of GitHub

GitHub should be described as:

- an operational layer
- a coordination surface
- a public preparation layer
- a helper-document surface
- a navigation layer into the project

GitHub should **not** be described as:

- the canonical authority layer
- the place where naming rights are granted
- the place where compliance is certified
- the constitutional layer of the project
- the final source of meaning

A safe formula is:

**GitHub explains and organizes.  
Canonical DOI documents define and constrain.**

---

## 9. The Correct Description of the Current Program Sequence

When describing project progression, use this logic:

1. External Layer-0
2. Internal Phase 0
3. Internal Phase 1
4. Internal Phase 2a
5. Internal Phase 2b
6. Lock 1
7. Lock 2
8. Broader Opening, if justified

A safe public sentence is:

**The program is currently structured as a kernel-first sequence in which broader opening comes only after internal phases and lock conditions have been satisfied.**

An unsafe sentence is:

**The competition is basically underway already.**

That sentence is too loose and creates false expectations.

---

## 10. What to Say About Participation

At the current stage, it is safe to say:

- “Readers can understand the structure and prepare bounded questions.”
- “Labs and engineers may evaluate whether they fit a narrow technical burden.”
- “The repository helps serious readers orient themselves before any broader opening.”
- “The project is visible enough to study, but not broadly open enough to overassume.”

It is **not** safe to say:

- “Anyone can now join as an official builder.”
- “This repository grants project status.”
- “This repository authorizes participation rights.”
- “This is already an open implementation program for the public.”

---

## 11. What to Say About Engineering Work

Safe language:

- “The engineering burden is still upstream.”
- “The main question is whether the kernel can survive drift, leak, cross-talk, and interpretation discipline.”
- “This is not a late-stage optimization problem; it is a structural measurement problem.”
- “The program is trying to distinguish signal from narrative.”

Unsafe language:

- “The engineering is basically done.”
- “This is just a packaging problem now.”
- “The hard part is finished.”
- “All that remains is productization.”

Those statements are too advanced for the actual stage.

---

## 12. What to Say About AI Governance

Safe language:

- “The project is relevant to AI-governance discussion because it asks whether state-level transformation can become more measurable.”
- “This is not presented as solved AI governance, but as an attempt to reduce blindness around state impact.”
- “The relevance to governance is conditional on whether the interface survives disciplined validation.”

Unsafe language:

- “This project solves AI governance.”
- “AI alignment is now technically solved.”
- “This repository proves governance is now complete.”

Those claims are too absolute.

---

## 13. What to Say About Consciousness

This is the most sensitive category.

Safe language:

- “The project does not claim a final theory of consciousness.”
- “The project is testing whether a bounded signal interface can be made reproducible.”
- “The work focuses on interface behavior, state-separability, and signal discipline.”
- “The repository should be read as a structured research effort, not as a declaration of metaphysical completion.”

Unsafe language:

- “Consciousness has now been fully measured.”
- “The nature of consciousness has been solved.”
- “This repository proves what consciousness is.”

The program may be ambitious.  
Its public language must still remain disciplined.

---

## 14. What to Say About Medical Meaning

Safe language:

- “This repository is non-diagnostic and non-therapeutic.”
- “No clinical meaning should be inferred from repository presence.”
- “Any future clinical pathway would require separate and much stricter work.”

Unsafe language:

- “This can diagnose patients.”
- “This can be used medically now.”
- “This is already relevant for patient care.”
- “This can be described as a treatment-related system.”

Do not pull medical language into a repository that is structurally upstream of medicine.

---

## 15. What to Say About Naming

Safe language:

- “Naming is constrained by canonical definitions and claims-governance rules.”
- “Readers should not assume that repository visibility grants naming rights.”
- “Similarity of concept does not itself justify public designation.”

Unsafe language:

- “Anything inspired by the repository can now be called Sal-Meter.”
- “GitHub documentation is enough to support designation.”
- “The public repository functions as naming permission.”

Naming is not a casual layer in this project.

---

## 16. What to Say About Compliance and Certification

Safe language:

- “This repository does not grant compliance, certification, or endorsement.”
- “Compliance language belongs to formal boundary conditions, not helper pages.”
- “Readers should not infer status from repository structure.”

Unsafe language:

- “This repository confirms compliance.”
- “A public build path implies certified status.”
- “The GitHub wording is enough to justify compliance claims.”

Repository presence is not a credential.

---

## 17. What to Say If You Need One Sentence Only

If you only have one sentence, use this:

**The Sal-Meter Kernel Program is a pre-opening, research-stage effort testing whether a bounded CAIS-aligned molecular-electrochemical interface can become reproducible enough to justify broader opening under strict authority and claims boundaries.**

That sentence is strong, accurate, and difficult to misuse.

---

## 18. What Public Speakers Should Avoid Most

The worst public errors are usually these:

- speaking too early
- speaking too absolutely
- speaking as if helper language were authority
- speaking as if the broader opening has already occurred
- speaking as if public visibility equals proof
- speaking as if conceptual ambition equals technical completion

A disciplined speaker must resist all of these.
