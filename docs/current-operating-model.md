## 1. Why This Page Exists

A frontier program becomes confusing very quickly when multiple surfaces begin speaking at once:

- canonical DOI records
- public foundation pages
- helper summaries
- GitHub repositories
- recruitment language
- PI-facing documents
- technical snapshots
- social posts

Without a clear operating model, people begin to misread:

- helper surfaces as authority
- research-stage work as product readiness
- early lab inquiry as open public participation
- technical feasibility as scientific proof
- public visibility as program maturity

This page exists to prevent that confusion.

Its purpose is simple:

**to state the present operating posture clearly enough that researchers, labs, technical contributors, and outside readers all understand what is actually live now.**

---

## 2. The Short Version

The current operating model is:

**kernel-first, pre-opening, research-stage, non-clinical, non-diagnostic, non-therapeutic.**

That means:

- the program is real
- the structure is being tightened
- the kernel is being stabilized
- selective technical routes are active
- broad public participation is **not** active
- public explanation is allowed
- canonical authority remains fixed outside GitHub

This is the right way to read the project now.

---

## 3. What the Program Is Trying to Do

At the highest level, the project is asking whether a bounded CAIS-aligned interface can be built strongly enough to support a future Sal-Meter pathway.

That burden does **not** begin with mass rollout or public competition.

It begins with smaller questions such as:

- can the molecular interface survive contact with matter honestly?
- can the electrochemical boundary behave stably enough to matter?
- can drift, perturbation, and repeatability be characterized cleanly?
- can preprocessing and validation avoid manufacturing false clarity?
- can the architecture remain bounded while the kernel is still fragile?

In other words:

**the current stage is not about acting as if everything is already open.  
It is about earning the right to move later with structure still intact.**

---

## 4. The Current Stage

The program should currently be read as a **pre-opening kernel program**.

That means the center of gravity is still inside:

- kernel stabilization
- interface reality
- evidence discipline
- reproducibility design
- terminology control
- claims boundary preservation
- controlled external inquiry rather than broad open entry

This is not a symbolic distinction.

It changes how every outside person should interpret the repository.

---

## 5. What Is Live Now

The following routes are genuinely active **now**.

## 5.1 ESL Recruitment

We are actively seeking an **Electrochemical Systems Lead (ESL)**.

This role is relevant to people who can seriously own issues such as:

- electrochemical measurement conditions
- electrode behavior
- bench stability
- drift characterization
- cartridge logic
- repeatability
- perturbation response
- SOP consistency
- acquisition reliability

The ESL is not a decorative title.

It is one of the structural leads needed to keep the kernel from becoming interpretive theater.

---

## 5.2 EStL Recruitment

We are actively seeking an **Evidence & Standardization Lead (EStL)**.

This role is relevant to people who can seriously own:

- metadata structure
- QC design
- leakage prevention
- validation integrity
- submission architecture
- reproducibility packaging
- version-aware evidence discipline
- cross-surface consistency

The EStL is essential because early programs often fail not only through weak chemistry or weak hardware,
but through weak evidence discipline.

---

## 5.3 External Layer-0 Feasibility Inquiry

Selective **External Layer-0** inquiry is active.

This is not broad public onboarding.
It is a bounded external path for labs or technical groups capable of helping reduce uncertainty in areas such as:

- iodine redox baseline characterization
- thiol / GSH-GSSG response mapping
- controlled perturbation response
- drift and repeatability behavior
- signal-interface survival
- chamber / cartridge boundary behavior

This route is vendor-facing and feasibility-facing.

It should not be mistaken for broader scientific endorsement or public program opening.

---

## 5.4 Aptamer G-Iodine Path

The **Aptamer G-Iodine** path is active as a core technical workstream.

This includes questions around:

- sequence feasibility
- interface compatibility
- molecular recognition structure
- signal-gating viability
- pathway design for a bounded measurement architecture

This path matters because it belongs to the kernel burden itself.

It is not merely a branding differentiator.
It is one of the reasons the architecture may or may not become technically real.

---

## 5.5 PI / Lab / Technical Lead Evaluation

PIs, laboratories, and technically serious teams may currently engage in a **bounded evaluation mode**.

That means:

- reading the current structure carefully
- identifying one ownable layer
- deciding whether a narrowly scoped pilot is realistic
- determining whether the current stage fits the lab’s burden tolerance

This is a serious route.

But it is still not the same as broad public participation.

---

## 6. What Is Not Live Now

The following are **not** currently active:

- broad public competition
- public SDK release
- public certification pathway
- compliance recognition
- product launch
- clinical use
- diagnostic use
- therapeutic positioning
- public language implying that kernel proof is already complete
- broad consumer or investor-facing rollout language

If a surface makes these things sound active now, that surface is ahead of the real program state.

---

## 7. The Program Order

The operating order is not arbitrary.

The intended order is:

1. external Layer-0 feasibility and interface reality
2. internal kernel-stage technical stabilization
3. controlled validation architecture
4. bounded Phase 0 to later execution sequence
5. internal review and lock logic
6. only then, later broader opening if earned

This means the present stage should always be read through a simple rule:

**kernel first, widening later.**

---

## 8. How GitHub Should Be Read

GitHub is a **public technical gateway** and **helper surface**.

It is useful for:

- meeting the right people early
- explaining current technical direction
- showing serious intent
- routing builders and labs to the right documents
- helping technically relevant outsiders recognize where they may fit

GitHub is **not**:

- the canonical authority layer
- a compliance grant surface
- a guarantee of participation
- a proof substitute
- a shortcut around DOI-bound structure

The repository is there to make the program legible.

It is not there to outrun the program.

---

## 9. Where Canonical Authority Lives

Canonical authority remains fixed in DOI-registered records and their associated public authority surfaces.

That means GitHub should be read together with:

- canonical DOI records
- boundary documents
- claims control documents
- structured public reading surfaces

When helper surfaces and canonical records differ in emphasis, the canonical boundary controls.

This must remain explicit.

## 11. The Best First Question for an Outside Team

The best first question is **not**:

- “How fast can we join?”
- “Is the full program already open?”
- “Can we present ourselves publicly as part of this?”
- “How quickly does this become a commercial platform?”

The best first question is:

**“What is one layer we can realistically own, test, or stabilize at the current stage?”**

That is the right entry posture.

---

## 12. The Best Outside Contribution Right Now

The best outside contribution right now is usually one of the following:

- reducing one technical uncertainty
- stabilizing one interface condition
- clarifying one leak pathway
- improving one reproducibility burden
- tightening one data or validation rule
- making one bounded pilot more real
- helping the kernel survive contact with disciplined evidence

The project needs fewer large symbolic gestures and more exact reductions of uncertainty.

---

## 13. Public Language That Fits the Current Stage

Good public language now includes phrases such as:

- research-stage
- kernel-first
- pre-opening
- non-clinical
- non-diagnostic
- non-therapeutic
- bounded feasibility
- interface validation
- reproducibility discipline
- early technical coordination
- PI / lab evaluation route
- external Layer-0 inquiry

Bad public language now includes phrases such as:

- already open competition
- proven consciousness device
- medical readiness
- diagnostic system
- certified compliance
- launch-ready product
- broad public participation is now active
- fully validated platform

Language drift is one of the easiest ways a frontier program loses structural integrity.

So wording must stay disciplined.

---

## 14. Legacy Terminology Note

Some earlier working materials may still contain legacy role names or earlier program labels.

The current public operating model uses the following lead roles:

- **ESL** = Electrochemical Systems Lead
- **EStL** = Evidence & Standardization Lead

The aptamer / SELEX path remains active as a technical workstream inside the present kernel-stage architecture.

Older language should not be treated as the latest operating model unless it has been explicitly carried forward.

---

## 15. If You Are a PI or Lab Lead

If you are a PI or lab lead, the correct reading posture is:

1. read the boundary first
2. identify one ownable layer
3. assess readiness honestly
4. decide whether a 3–6 month bounded pilot is realistic
5. engage only if the fit is real

The most useful labs are not necessarily the biggest.
They are often the labs that can hold one burden clearly.

---

## 16. If You Are a Technical Contributor

If you are a technical contributor, the correct reading posture is:

1. do not assume the whole stack must be yours
2. do not treat helper language as proof
3. ask whether your actual skill reduces one present uncertainty
4. respect the canonical/helper distinction
5. stay inside the public claims boundary

This program benefits most from people who strengthen the structure without inflating it.

---

## 17. Final Reading Rule

If you remember only one thing from this page, remember this:

**The program is live now at the level of kernel stabilization, lead recruitment, bounded external feasibility, and PI / lab evaluation.  
It is not yet live as a broad public opening.**

That distinction protects the architecture.

It also protects the people entering it.
