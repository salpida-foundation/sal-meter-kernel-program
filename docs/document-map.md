# Document Map — What to Read, What to Use, What to Ignore

> This page explains how to navigate the repository without confusing active entry documents, operational helper pages, Korean guidance pages, and older working documents.  
> It is a reading map, not a source of canonical authority.

---

## 1. Why This Page Exists

As a repository grows, it becomes harder for readers to tell the difference between:

- current entry documents
- operational helper pages
- Korean guidance pages
- older working documents
- legacy files that still exist but are no longer the main path

Without a map, visitors often read the wrong file first.

This page exists to prevent that.

Its role is simple:

**to show which documents are active, which are secondary, and which should be treated as legacy or low-priority unless specifically needed.**

## 6. Legacy or Lower-Priority Working Documents

Some documents in the repository may still exist because they were useful during earlier drafting, working, or transitional stages.

Examples may include files such as:

- `00_philosophy_foundation.md`
- `01_technical_details.md`
- `02_phase_roadmap_en.md`
- `03_team_composition.md`
- `04_faq_extended_en.md`
- `05_application_guide_en.md`

These files are not necessarily “wrong,” but they should **not** be treated as the default first path unless they are explicitly upgraded, re-linked, or intentionally reused.

### How to read these files
Treat them as one of the following unless clearly reactivated:

- working notes
- earlier helper drafts
- extended support material
- secondary reference material
- legacy documents awaiting either revision or retirement

### Practical rule
If a document is not linked clearly from the active core path, it should not be assumed to be part of the current primary reading route.

---

## 7. Active vs. Legacy Rule

A simple rule helps:

### Active documents
Documents are active if they do one or more of the following:

- are linked directly from `README.md`
- are part of the current recommended reading path
- clarify current status, structure, or engagement
- reduce confusion for new readers
- fit the current kernel-first project direction

### Legacy / lower-priority documents
Documents should be treated as legacy or lower-priority if they:

- belong to earlier drafts or earlier naming stages
- are not linked from the current main path
- duplicate newer pages
- contain extended detail that is not needed for first entry
- have not yet been upgraded to the current structure

---

## 8. What New Readers Should Ignore for Now

A new reader should ignore, for now:

- raw file-number order
- old draft sequencing
- extended detail pages not linked from the core path
- any document that looks technical but is not clearly part of the current reading route
- repository sprawl mistaken for current authority

In other words:

**do not assume every file deserves equal reading priority.**

## 10. Which Documents Carry Canonical Authority?

None of the GitHub helper pages carry canonical authority by themselves.

GitHub pages can:

- explain
- orient
- summarize
- map
- clarify

But canonical authority remains only in DOI-registered documents.

That means:

- GitHub pages = operational layer
- DOI documents = authoritative layer

This page is only a navigation tool.

It does not override canon.

---

## 11. How to Use This Repository Correctly

Use the repository like this:

- start with active entry documents
- use Korean pages only as support, if needed
- use lower-priority documents only when a specific reason exists
- do not assume file presence equals active status
- move into DOI documents when authority, naming, compliance, or meaning matters

That is the safest reading discipline.

---

## 12. What Should Eventually Happen to Older Pages

Older pages should eventually be handled in one of four ways:

1. **upgrade**  
   If still useful, rewrite them into the new structure

2. **merge**  
   If duplicated by better pages, merge key content into active pages

3. **archive mentally**  
   Leave them present but treat them as low-priority working material

4. **retire**  
   If no longer useful, stop linking to them and let them fade from the active path

The goal is not to erase the past.

The goal is to keep the current path readable.

---

## 13. One-Line Summary

This repository should not be read as a flat folder of equal files.

It should be read as a **layered document system**:

- active core pages first
- Korean support pages second
- older working files only when specifically needed
- DOI authority last and highest

## 14. Final Position

A strong repository is not one that contains many documents.

It is one where readers know exactly where to start, where to go next, and what not to confuse with authority.

That is the purpose of this page.
