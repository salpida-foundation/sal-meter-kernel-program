# 자주 묻는 질문 — Sal-Meter Kernel Program

> 이 문서는 한국어 사용자가 README, quick-start, status 문서를 읽은 뒤 가장 자주 가지는 질문에 빠르게 답하기 위한 FAQ입니다.  
> 정본 권위는 GitHub가 아니라 **DOI 등록 문서 (Zenodo / OSF)** 에 있습니다.

---

## 1. 이 저장소는 무엇인가요?

이 저장소는 **Sal-Meter Kernel Program**의 공개 운영 인덱스입니다.

즉, 이곳은:

- 프로젝트의 현재 상태를 설명하고
- helper 문서로 빠르게 연결하며
- FAQ와 상태 문서를 통해 오해를 줄이고
- broader opening 이전 단계에서 구조와 방향을 정리하는

**운영·안내·조정 레이어**입니다.

정본 권위 문서 자체는 아닙니다.

---

## 2. 지금 공개 competition이 이미 열린 상태인가요?

**아닙니다.**

현재 이 저장소는 **broad public competition이 열린 상태가 아닙니다**.

지금은 **kernel-first 단계**입니다.

즉,

- External Layer-0
- Internal Phase 0
- Internal Phase 1
- Internal Phase 2a
- Internal Phase 2b
- Lock 1
- Lock 2

를 거친 뒤에야, 조건이 충족될 경우 broader opening이 검토됩니다.

저장소가 살아 움직인다고 해서  
곧바로 competition이 열린 것은 아닙니다.

---

## 3. Kernel Program과 Broader Opening은 무엇이 다른가요?

차이는 매우 큽니다.

### Kernel Program
핵심 측정 커널이 실제로 존재하는지, 반복 가능한지, 의미가 있는지를 먼저 검증하는 단계입니다.

### Broader Opening
커널이 충분히 고정된 뒤에야 외부 빌더, 더 넓은 참여, 비독점 확장 경로를 검토하는 단계입니다.

한 줄로 줄이면:

- **Kernel Program** = 먼저 증명
- **Broader Opening** = 그 다음 확장

---

## 4. External Layer-0는 무엇인가요?

External Layer-0는 **별도의 pre-phase feasibility 지원 트랙**입니다.

이 단계의 질문은 비교적 단순합니다.

**iodine redox + thiol/disulfide 인터페이스가 실제로 안정적이고 반복 가능한 신호 창을 만들 수 있는가?**

즉, Layer-0는 초기 가능성 탐색 단계이며,  
그 자체가 broader opening과 동일한 것은 아닙니다.

---

## 5. 이 저장소는 의식을 이미 측정했다고 주장하나요?

**아닙니다.**

이 저장소는 “의식이 이미 완전히 측정되었다”는 선언을 하지 않습니다.

오히려 더 제한된 질문을 다룹니다.

**통제된 상태 변화에서, 분자-전기화학 인터페이스가 반복 가능한 신호 이벤트를 만들 수 있는가?**

즉, 이것은 최종 증명 선언이 아니라  
**구조화된 시험의 시작**입니다.

---

## 6. Sal-Meter는 의료기기인가요?

현재 이 저장소 범위에서는 **아닙니다**.

지금 단계의 기본 경계는 다음과 같습니다.

- 연구용
- 비진단적
- 비치료적
- 비임상 의미

즉, 이 저장소의 문구를

- 진단
- 치료
- 의료기기 승인
- 임상 readiness

로 읽으면 안 됩니다.

---

**아닙니다.**

이 저장소는 다음을 자동으로 뜻하지 않습니다.

- certification
- endorsement
- 공식 authorized-user status

- mark usage right

GitHub 저장소는 인증서가 아닙니다.

## 8. GitHub가 정본인가요?

**아닙니다.**

정본 권위는 GitHub에 있지 않습니다.

정본 권위는 **DOI 등록 문서 (Zenodo / OSF)** 에만 있습니다.

GitHub는 이 프로젝트에서 다음 역할을 합니다.

- operational index
- public preparation layer
- coordination surface
- discussion space
- orientation layer

즉,

- GitHub = 운영 레이어
- DOI 문서 = 권위 레이어

입니다.

---

## 9. helper 문서는 믿어도 되나요?

믿을 수는 있지만, **권위 문서로 읽으면 안 됩니다**.

helper 문서는 다음 용도로 좋습니다.

- 빠른 진입
- PI 의사결정
- 엔지니어링 개요 파악
- 프로그램 구조 이해
- FAQ / 상태 정리

하지만 helper 문서는 정본 문서를 대체하지 않습니다.

helper 문서는 **입구**이고,  
canonical DOI 문서는 **권위**입니다.

지금 단계에서 임의로 그렇게 해석하면 안 됩니다.

“Sal-Meter”라는 명칭은  
단순한 유사성이나 분위기가 아니라,  
**정의·경계·명칭 규율**과 연결되어 있습니다.

즉,

- 비슷해 보인다고 해서
- 아이디어를 참조했다고 해서
- GitHub를 읽었다고 해서

곧바로 그 명칭을 정당하게 쓸 수 있다고 보면 안 됩니다.

“당장 넓게 참여”보다

**“검증을 견딜 수 있는가?”** 라면  
이 저장소는 의미가 큽니다.

## 14. 왜 저장소 톤이 이렇게 엄격한가요?

일부러 그렇습니다.

프로젝트를 빨리 열면 흔히 다음 문제가 생깁니다.

- 개념 드리프트
- 약한 엔지니어링 위의 큰 서사
- premature commercialization
- naming confusion
- helper 문서와 canonical authority의 혼동
- 측정 전에 의미를 과장하는 오류

이 저장소의 엄격함은  
닫기 위한 엄격함이 아니라  
**커널을 살리기 위한 엄격함**입니다.

## 15. negative result도 의미가 있나요?

**아주 중요합니다.**

이 프로젝트에서는 positive result만 가치 있는 것이 아닙니다.

예를 들어 이런 것도 충분히 가치 있습니다.

- 경로가 재현성 없이 무너짐
- leak source가 드러남
- preprocessing 가정이 과장되었음이 밝혀짐
- state-separability claim이 stricter test에서 무너짐
- 설계상 false path가 정리됨

애매한 yes보다  
깨끗한 no가 더 강할 수 있습니다.

## 16. 영어와 한국어는 각각 어떤 역할인가요?

이 저장소는 **영문 메인 / 한글 안내 레이어** 구조를 따릅니다.

원칙은 다음과 같습니다.

- 영어 = 국제 과학 접근성을 위한 메인 언어
- 한국어 = 빠른 이해를 위한 지원 레이어
- GitHub prose = operational layer
- DOI 문서 = authoritative layer

즉,

- 영문 = 공용 진입면
- 한글 = 안내층
- 정본 = DOI

한글 문서는 도움이 되지만,  
정본 권위를 따로 만들지 않습니다.

## 17. 한국어 문서는 공식 문서인가요?

한국어 문서는 **공식 안내 문서**로서 의미는 있지만,  
**별도의 정본 권위 레이어**는 아닙니다.

즉, 한국어 문서는 운영상 유용한 support layer이지,  
DOI canonical authority를 대체하지 않습니다.

곧바로 “열렸다”는 뜻은 아닙니다.
